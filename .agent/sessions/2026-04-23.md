### Diagnosis & Pembersihan PDF Corrupt

**Root cause ditemukan:** PDF di `int/`, `gen/`, `bas/` rusak karena binary file ditransfer sebagai UTF-8 text. Setiap byte non-ASCII (> 0x7F) di-replace dengan UTF-8 replacement character `\xef\xbf\xbd` (U+FFFD). Ini merusak seluruh compressed stream content di dalam PDF.

**Indikator korupsi:** `\xef\xbf\xbd` muncul di 20 byte pertama header PDF (di mana seharusnya ada binary comment bytes `%âãÏÓ`).

**Yang dihapus:**
- `int/` — semua 13 file (0/13 extractable, semua corrupt)
- `gen/` — semua 80 file (0/80 extractable, mix corrupt + image-based)
- `bas/` — semua 7 file (0/7 extractable, atlas bergambar besar + corrupt)
- `obg/buku-saku-pelayanan-kesehatan-ibu.pdf` — corrupt, 0 pages
- `obg/53panduan-penatalaksanaan-kasus-obstetri.pdf` — corrupt; 5 chunks tersimpan tapi isinya MuPDF error messages (garbage), bukan medical content

**Yang bersih:** `pha/` dan `ped/` — PDF 1.2 standar, tidak ada korupsi.

### Bug Fix: Chunker + PyMuPDF stdout

**Bug 1 — Chunker menghasilkan 1 chunk besar:**
PDF teks tidak punya markdown headings (`# ## ###`). Chunker lama split by `\n\n+` (double newline) saja. PyMuPDF output pakai single `\n`, sehingga seluruh teks dianggap 1 section raksasa → 1 chunk 35k+ chars → Ollama embedding gagal (token limit).

**Fix:** Tambah fallback ke line-grouping: kalau `\n\n+` split menghasilkan ≤1 paragraph, group lines satu per satu sampai mencapai `MAX_TOKENS=800`.

**Bug 2 — MuPDF error messages masuk ke DB:**
Di Windows, PyMuPDF menulis C-level error output ke fd 1 (stdout) bukan stderr. `execFileAsync` capture stdout → error messages ikut masuk sebagai chunk content.

**Fix:** `fitz.TOOLS.mupdf_display_errors(False)` di awal `pdf_extract.py`.
