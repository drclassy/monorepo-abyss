Codex IDE Field Manual

Codex Covernant

Handbook ini merangkum cara paling sehat memakai Codex langsung di editor seperti VS Code, Cursor, dan editor kompatibel lain: bagaimana menyusun prompt, kapan memakai plan, kapan naik ke cloud, bagaimana menyalakan skill dan plugin, model mana yang paling masuk akal, dan anti-pattern apa yang paling sering bikin agent terasa "ngawur".

Fokus halaman ini adalah penggunaan harian di IDE, dengan pondasi resmi dari dokumentasi OpenAI dan snapshot kemampuan yang memang tersedia di sesi Codex saat halaman ini dibuat pada 30 Mei 2026.

Principles

Prinsip dasar supaya Codex terasa seperti teammate, bukan mesin tebak

Hampir semua hasil buruk datang dari konteks yang kabur, izin yang kelewat longgar, atau definisi selesai yang tidak eksplisit.

Layered instructions Biarkan AGENTS bekerja berlapis
  • Codex membaca ~/.codex/AGENTS.md dulu, lalu file project dari root sampai folder kerja.
  • File yang lebih dekat ke direktori aktif override aturan yang lebih umum.
  • Taruh instruksi spesifik sedekat mungkin ke area kerja yang memang membutuhkannya.
Prompt anatomy Pakai empat blok inti di prompt
  • Goal: apa yang mau dibangun atau diubah.
  • Context: file, error, docs, caller, atau screenshot yang penting.
  • Constraints: style, safety, boundary, atau standar repo.
  • Done when: kondisi selesai yang bisa diverifikasi.
Plan first Untuk tugas sulit, planning lebih murah daripada rework
  • OpenAI merekomendasikan plan dulu untuk task kompleks, ambigu, atau susah dijelaskan.
  • Di IDE, /plan atau pola plan mode cocok untuk membangun batas kerja sebelum eksekusi.
Deliberate autonomy Pilih izin dan network dengan sengaja
  • workspace-write adalah mode rendah friksi yang sehat untuk kerja lokal.
  • Network tetap mati secara default pada baseline ini kecuali Boss menyalakannya di konfigurasi.
  • danger-full-access hanya masuk akal saat memang mau melepas semua boundary.
Clean context Jaga thread utama tetap bersih
  • Subagent berguna untuk eksplorasi, tes, triage, dan summarization yang read-heavy.
  • Parallel writer di file yang sama justru bikin konflik dan overhead koordinasi.
Verify always Jangan tutup task pakai narasi doang
  • Workflow resmi Codex selalu menaruh verifikasi sebagai bagian eksplisit dari loop kerja.
  • Minimal: repro ulang, test terkecil yang relevan, lalu review diff sebelum handoff.
Workflow

Loop harian yang paling waras di IDE

Ini pola yang paling konsisten: eksplorasi cepat di IDE, eksekusi dengan prompt yang punya done-criteria, lalu review dan verifikasi sebelum keluar.

Open Context

Buka file yang paling relevan

Panduan workflow resmi menekankan bahwa IDE extension otomatis memasukkan file terbuka sebagai konteks. Jadi buka caller, target file, dan test terdekat dulu.

Attach Precisely

Tambahkan konteks selektif

Pilih blok kode lalu gunakan Add to Codex Thread, atau tag file dengan @path. Lebih baik tiga file yang tepat daripada tiga puluh file yang asal.

Prompt Well

Kasih Goal, Context, Constraints, Done when

Kalau bug bisa direpro, sertakan resep reproduksi. Kalau UI, sertakan screenshot plus batas implementasi seperti framework, route, dan requirement mobile.

Pick Mode

Pilih local, cloud, atau review mode

Pakai /local untuk kerja di workspace saat ini, /cloud untuk pekerjaan panjang yang ingin diparalelkan, dan /review sebelum commit.

Run Small Loops

Iterasi kecil menang atas prompt raksasa

Untuk UI, minta 2-3 opsi lalu pilih satu. Untuk refactor besar, mulai dengan plan atau milestone, bukan "rewrite semuanya".

Close Cleanly

Verifikasi, review, baru handoff

Minta Codex menjalankan check terkecil yang relevan, lalu pakai /status bila perlu untuk melihat konteks dan limit thread sebelum lanjut.

Tips & Tricks

Kebiasaan kecil yang efeknya besar

Banyak peningkatan kualitas Codex datang dari ergonomi: shortcut, slash command, tagging file, dan pemilihan reasoning effort yang pas.

@file references Tag file langsung di prompt
  • Contoh resmi: @example.tsx dan @resources.ts untuk menambah halaman baru dengan referensi yang jelas.
  • Ini membuat prompt lebih pendek tapi tetap tepat sasaran.
Reasoning effort Mulai dari medium, naik hanya bila perlu
  • Di IDE, switcher model juga mengatur reasoning effort.
  • Low untuk task sempit, Medium default, High untuk debugging atau perubahan rumit.
Slash commands Biasakan hidup di chat input
  • /auto-context untuk konteks IDE otomatis.
  • /status untuk thread ID, usage, dan rate limit.
  • /review untuk review working tree atau diff.
  • /goal untuk objective yang persisten.
Key bindings Hilangkan friction di command palette
  • chatgpt.newChat sudah default ke Ctrl+N di Windows/Linux.
  • Bind juga chatgpt.addToThread dan chatgpt.addFileToThread kalau Boss sering kerja selection-based.
Cloud delegation Lokal untuk desain, cloud untuk long-run
  • Dokumen workflow resmi menyarankan desain hati-hati secara lokal lalu delegasi milestone implementasi ke cloud.
  • Bagus untuk refactor panjang yang tidak perlu terus memenuhi thread utama.
Permissions hygiene Network dan full access itu keputusan, bukan default
  • Secara lokal, network mati pada mode default sampai diaktifkan di konfigurasi.
  • Kalau perlu network, lebih aman batasi juga lewat proxy domain policy bila workflow mengizinkan.

Anti-Patterns

Pola yang paling sering membuat Codex tampak "kurang pintar"

Biasanya masalahnya bukan modelnya, tetapi cara kita memberi konteks, izin, dan scope.

Prompt kabur Tidak menyebut Goal, Constraints, atau Done when
  • Codex jadi menebak standar keberhasilan sendiri.
  • Hasilnya sering terlihat "bagus" tetapi salah boundary.
Single giant thread Semua hal dipaksa masuk satu percakapan panjang
  • Ini resep context rot: thread utama dipenuhi log, eksperimen, dan asumsi lama.
  • Mulai thread baru saat fase kerja benar-benar berganti.
Auto-context abuse Membuka terlalu banyak file dan berharap agent memilah sendiri
  • Auto context membantu, tapi tetap perlu seleksi manusia.
  • Buka file yang relevan, bukan seluruh repo.
Premature full access Langsung menyalakan network atau full access tanpa alasan
  • Default lokal sudah cukup untuk banyak task coding biasa.
  • Naikkan izin hanya saat bottleneck-nya memang boundary teknis.
Write-heavy parallelism Banyak subagent mengedit area yang sama
  • OpenAI menyarankan subagent paralel terutama untuk eksplorasi, tes, triage, dan summarization.
  • Penulisan paralel di file yang sama sering menambah konflik, bukan kecepatan.
No verification Menutup task hanya dari ringkasan agent
  • Selalu minta command, repro, atau review yang membuktikan perubahan.
  • Kalau tidak bisa diverifikasi, tulis jelas risiko yang masih tersisa.
Snippets

Prompt pattern yang efektif dipakai langsung di IDE

Gunakan sebagai starter. Kuncinya bukan panjang prompt, tapi kejelasan goal dan verifikasinya.

Onboarding codebase Memahami area baru dengan cepat 
Goal: pahami request flow modul ini.
Context: @route.ts @service.ts @schema.ts
Constraints: jangan ubah file.
Done when: beri saya ringkasan alur, validasi penting, dan 2 gotcha perubahan.
Bugfix dengan repro Memaksa loop perbaikan tetap objektif 
Bug: tombol Save menampilkan sukses tapi perubahan tidak tersimpan.
Repro:
1. Jalankan app
2. Buka /settings
3. Toggle Enable alerts
4. Click Save
5. Refresh, state kembali
Constraints: jangan ubah API shape; fix minimal; tambah regression test bila masuk akal.
Done when: repro sudah hijau dan jelaskan cara verifikasinya.
UI iteration Meminta opsi dulu sebelum memilih arah 
Review halaman landing ini dan usulkan 3 peningkatan visual.
Jaga struktur konten tetap sama.
Setelah itu tunggu saya pilih satu arah sebelum mengubah kode.
Plan-first Untuk pekerjaan yang rumit atau masih kabur 
/plan
Kita mau memecah auth subsystem menjadi parsing token, session loading,
dan permissions. Tolong susun plan, file yang mungkin disentuh, risiko,
dan verifikasi sebelum implementasi.
Skill invocation Memaksa workflow tertentu 
$openai-docs
Cari panduan resmi terbaru OpenAI untuk model yang tepat,
lalu rangkum rekomendasi teknis yang relevan untuk tugas ini.
Pre-commit review Second set of eyes sebelum push 
/review Focus on regressions, edge cases, and security concerns.
Setelah itu ringkas temuan berdasarkan severity, lalu beri saran verifikasi.
Models

Model selection untuk Codex di IDE

Per dokumentasi Codex resmi yang current pada 30 Mei 2026, pemilihan model sebaiknya sederhana: satu default kuat, satu opsi ringan, dan satu lane coding-native untuk kasus khusus.

Default best pick gpt-5.5

OpenAI merekomendasikan model ini untuk mayoritas task Codex. Paling kuat untuk coding kompleks, computer use, knowledge work, dan research workflow.

Gunakan untuk: task utama Cocok untuk: planning, debugging, synthesis
Fast / lighter lane gpt-5.4-mini

Opsi cepat dan lebih hemat untuk tugas ringan, iterasi kecil, atau subagent yang hanya perlu bounded reasoning.

Gunakan untuk: quick edits Cocok untuk: triage, ref kecil, subagents
Coding-native gpt-5.3-codex

Model coding khusus dengan kemampuan software engineering yang sangat kuat. Dokumentasi resmi bahkan menyebut kemampuan coding-nya kini juga menjadi tenaga inti di balik gpt-5.4.

Gunakan untuk: engineering berat Cocok untuk: refactor kompleks, code reasoning
Near-instant preview gpt-5.3-codex-spark

Research preview untuk iterasi coding hampir instan. Lebih cocok buat loop cepat, bukan default universal. Tersedia untuk ChatGPT Pro pada snapshot docs ini.

Gunakan untuk: realtime iteration Catatan: preview / availability terbatas
Reasoning effort Atur kedalaman berpikir, bukan hanya nama model
  • Low untuk task yang sudah sangat terdefinisi.
  • Medium sebagai default aman untuk mayoritas pekerjaan IDE.
  • High untuk debugging rumit, root-cause analysis, atau perubahan lintas modul.
Practical heuristic Heuristik cepat memilih model
  • Kalau Boss ragu: mulai di gpt-5.5 / medium.
  • Kalau task terlalu kecil untuk model berat: turun ke gpt-5.4-mini.
  • Kalau butuh fokus engineering yang kuat: uji gpt-5.3-codex.
