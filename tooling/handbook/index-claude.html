Core Principles

Prinsip fundamental untuk kerja lebih efektif dengan Claude

Enam prinsip ini membentuk foundation dari workflow yang healthy: skill-driven, design-first, systematic debugging, dan verification sebelum ship.

Think First, Code Second

Brainstorm sebelum implementation — skill harus di-invoke sebelum baris pertama kode ditulis. Creative work wajib melalui design cycle.

Skill: superpowers:brainstorming
Skills Override Defaults

Jika skill tersedia, gunakan. Bukan pilihan, bukan opsional — bahkan 1% kemungkinan skill berlaku adalah trigger untuk invoke.

Minimal viable solution dimulai dari skill yang tepat.
One Question at a Time

Jangan banjiri user dengan pertanyaan — satu per pesan. Ini mencegah cognitive overload dan membuat brainstorming lebih focused.

Multiple choice preferred untuk clarity, tapi satu saja per turn.
Advisor Before Commitment

Panggil `advisor()` sebelum langkah substansial, bukan setelah. Ini membuat keputusan approach lebih terdokumentasi dan durable.

Conversation history tersimpan — gunakan untuk reference setelah session berakhir.
Prefer Existing Patterns

Cari fungsi/utilitas yang sudah ada sebelum menulis baru. YAGNI ruthlessly — solve immediate problem, extend when needed.

Reuse patterns dari skill atau existing codebase sebelum abstraksi baru.
Durable Over Ephemeral

Simpan hasil kerja ke file/commit sebelum advisory — session bisa berakhir kapan saja. Plan file, spec docs, dan commit messages adalah contracts.

Memory system ada untuk context lintas session — dokumentasikan lessons learned.
Daily Workflow

Urutan kerja dari start sampai ship

Six-step workflow yang health untuk repo nyata: Session Setup, Skill Check, Design, Implement, Debug, Ship. Setiap step punya entry point dan exit criteria yang jelas.

SESSION START

Invoke using-superpowers

Dibaca otomatis setiap sesi — establishes skill discovery rules dan mengingatkan work mode utama. Ini adalah "warm-up" sebelum apa pun.

CHECK SKILLS

Scan applicable skills

Bahkan 1% kemungkinan skill berlaku → invoke dulu. Skills tell HOW to approach; jangan skip hanya karena terasa simple atau immediate.

DESIGN

brainstorming → spec → writing-plans

Creative work wajib melalui brainstorm; design spec → plan sebelum kode. Ini prevents rework dan membuat scope lebih clear untuk implementation.

IMPLEMENT

TDD / executing-plans

Test dulu, kode kemudian. Gunakan subagent-driven-development untuk task paralel; prefer parallel agents untuk independent work.

DEBUG

systematic-debugging

Setiap bug/test failure → debugging skill sebelum propose fix. Systematic approach prevents blind guessing dan documents root cause.

SHIP

finishing-a-development-branch

Merge, PR, atau cleanup — guided structured options. Verification closes loop: build, test, focused lint sebelum close-out.

Tips & Tricks

Power-user habits yang bikin Claude lebih efisien

Kebiasaan ini paling terasa efeknya di hari ke hari: lebih sedikit prompt berulang, lebih sedikit approval melelahkan, dan lebih tajam saat memilih tool.

/code-review ultra Deep Cloud Review
  • Multi-agent cloud review untuk PR besar atau refactor kompleks.
  • Pakai saat perlu perspektif mendalam dari berbagai sudut (architecture, performance, security).
  • Billed per run — gunakan strategis, bukan di setiap PR trivial.
/code-review --fix Auto-Apply Fixes
  • Review + langsung apply ke working tree; medium/high effort untuk coverage lebih luas.
  • Useful untuk cleanup, simplification, dan efficiency improvements saat ready to commit.
  • Review dulu sebelum apply jika ada risiko behavioral change.
/loop 5m /command Recurring Tasks
  • Jalankan command berulang setiap N menit; omit interval = self-paced polling.
  • Great untuk long-running checks: deploy monitoring, status polling, log tailing.
  • Returns single notification saat done, atau periodic updates sesuai command output.
advisor() Expert Second Opinion
  • Lihat seluruh conversation history; panggil sebelum commit approach ke git.
  • Useful untuk architecture review, design validation, dan risk gate sebelum big change.
  • Creates documented record — reference setelah session berakhir via memory system.
/schedule Cron Agents
  • Remote agent yang jalan terjadwal; satu-kali atau recurring pada cron interval.
  • Useful untuk nightly jobs, weekly syncs, atau automated checks yang consistent.
  • Limited to durable artifacts (e.g., flags with cleanup dates, instrumentation with "remove after X").
agent-browser Browser Automation
  • Web interaction, form fill, screenshot, Electron app automation — e.g., VS Code, Slack, Discord.
  • Use untuk exploratory testing, dogfooding, QA, bug hunts, atau reviewing app quality.
  • Also for Vercel Sandbox microVMs dan AWS Bedrock AgentCore cloud browsers.

Anti-Patterns

Red flags yang sering bikin workflow ngaco

Ini adalah "thoughts yang bikin overthinking" — setiap satu adalah signal untuk check skill applicable atau change approach.

"This is just a simple question"

Reality: Questions are tasks. Bahkan simple questions sering butuh skill untuk answer dengan baik.

→ Check for skills BEFORE first response
"I need more context first"

Reality: Skill check comes BEFORE clarifying questions. Skills tell HOW to gather context — jangan skip dulu.

→ Invoke skill dulu, context gathering baru setelah
"Let me explore the codebase first"

Reality: Skills tell HOW to explore. Undirected exploration buang waktu — use skill untuk efficient discovery.

→ Use Explore subagent atau skill-guided investigation
"I remember this skill"

Reality: Skills evolve. Training data != current skill version. Baca current skill content dari CLI dulu.

→ Always invoke Skill tool; jangan rely on memory
"This is too simple to need a design"

Reality: Every project goes through design. "Simple" projects paling sering bikin assumption errors. Design bisa short, tapi MUST present.

→ Even todo list wajib design cycle
"I'll just do this one thing first"

Reality: Check BEFORE doing anything. Undisciplined action = wasted rework. Skills prevent this.

→ Skill check is prerequisite, not nice-to-have
Skill Workflows

Common workflow snippets using skills

Ini bukan template sakti, tapi urutan skill yang common dan biasanya effective untuk different scenarios.

PLANNING Start a Feature
  • brainstorming → spec write → writing-plans → executing-plans
  • Workflow ini paling sehat untuk feature baru atau significant refactor.
  • Spec file stored di docs/superpowers/specs/ untuk reference post-session.
REVIEW Code Review Flow
  • /code-review low/medium/high untuk iterative feedback;
  • /code-review ultra untuk deep multi-agent review (billed);
  • superpowers:requesting-code-review sebelum push.
DEBUGGING Debug Workflow
  • systematic-debugging → diagnose → fix → verify → commit
  • Never guess; always root-cause analysis dulu.
  • Documentation step: commit message menjelaskan WHY, bukan WHAT.
SCHEDULING Automate Tasks
  • /loop N interval /command untuk recurring checks;
  • /schedule untuk cron agents dengan durable artifacts;
  • Jangan gunakan untuk ephemeral task — gunakan untuk long-running atau event-driven.
Claude Models

3 Claude Models tersedia

Choose model berdasarkan complexity: Opus untuk reasoning mendalam, Sonnet untuk balanced work, Haiku untuk quick tasks. Default saat ini adalah Sonnet 4.6.

Model ID Speed Best For
Claude Opus 4.8 claude-opus-4-8 Slower Most complex tasks, deep reasoning, ultra code review, architecture planning
Claude Sonnet 4.6 claude-sonnet-4-6 Balanced Default production work, feature implementation, most tasks (current model)
Claude Haiku 4.5 claude-haiku-4-5-20251001 Fastest Simple tasks, commit messages, quick answers, triage, lightweight operations
/fast Mode Faster Output from Stronger Model
  • Menggunakan Opus dengan output lebih cepat — tidak downgrade ke model lebih kecil.
  • Useful saat butuh reasoning depth tapi response speed juga penting.
  • Available di Opus 4.8 / 4.7 / 4.6; toggle dengan /fast command.
Model Selection How to Choose
  • Start dari Auto-select jika uncertain — biar system pick optimal model.
  • Explicit choice saat tahu task complexity level (simple → Haiku, complex → Opus).
  • Cost scales dengan model — prefer Haiku untuk triage, Sonnet untuk production.
