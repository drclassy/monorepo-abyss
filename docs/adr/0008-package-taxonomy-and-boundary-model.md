# ADR 0008 — Package Taxonomy and Boundary Model

**Date:** 2026-04-30  
**Status:** Accepted  
**Decider:** Dr. Ferdi Iskandar (Chief)  
**Mission:** ABYSS-REPO-STRUCTURE-001

---

## Context

The monorepo `packages/` directory uses a flat layout where all packages — proprietary crown-jewel AI engines, platform infrastructure, clinical knowledge substrate, shared primitives, and developer tooling — sit at the same level. This creates:

- No architectural signal for human developers or AI coding agents
- No enforced dependency direction (platform importing sentra is structurally possible)
- No immediate visual distinction between IP-critical code and generic utilities
- Risk of future agents creating packages in wrong locations without guidance

This ADR establishes the canonical taxonomy, dependency direction, and enforcement model for the `packages/` directory.

---

## Decision

The `packages/` directory is restructured into five semantic category folders:

```
packages/
  sentra/     ← Proprietary Sentra crown-jewel capabilities
  platform/   ← Runtime infrastructure substrate
  clinical/   ← Clinical knowledge and safety substrate
  shared/     ← Low-level reusable primitives
  tooling/    ← Developer and build tooling
```

### Category Definitions

| Category | Folder | Definition |
|---|---|---|
| Sentra Crown Jewel | `packages/sentra/*` | Proprietary product capability, core Sentra domain logic, business-differentiating intelligence. IP-critical. |
| Platform Substrate | `packages/platform/*` | Runtime infrastructure, persistence, orchestration, ingestion, vector/RAG infrastructure. |
| Clinical Substrate | `packages/clinical/*` | Clinical knowledge, clinical safety rules, guidelines, FHIR, pharmacotherapy, medical references. |
| Shared Primitives | `packages/shared/*` | Low-level reusable types, UI primitives, tokens, common utilities. |
| Tooling | `packages/tooling/*` | Build tooling, lint config, TypeScript config, generators, internal validation tools. |

### Invariant Rules

1. **Group folders are not packages.** `packages/sentra/`, `packages/platform/`, `packages/clinical/`, `packages/shared/`, `packages/tooling/` must never contain `package.json`.

2. **New packages must be created inside a category folder.** Direct children of `packages/` are forbidden unless explicitly classified as a special case by Chief.

3. **Package names (`package.json` `name` field) do not change** when packages are placed inside taxonomy folders.

---

## Dependency Direction Rule

The intended dependency direction is unidirectional top-down:

```
apps
  ↓
packages/sentra
  ↓
packages/platform  +  packages/clinical
  ↓
packages/shared
```

`tooling` is special-purpose and not imported by runtime packages.

### Allowed

```
apps/*              → packages/sentra/*
apps/*              → packages/platform/*
apps/*              → packages/clinical/*
apps/*              → packages/shared/*

packages/sentra/*   → packages/platform/*
packages/sentra/*   → packages/clinical/*
packages/sentra/*   → packages/shared/*

packages/platform/* → packages/shared/*
packages/clinical/* → packages/shared/*

packages/tooling/*  → packages/shared/*
```

### Forbidden (Architectural Inversion)

```
packages/shared/*   → packages/platform/*
packages/shared/*   → packages/clinical/*
packages/shared/*   → packages/sentra/*

packages/platform/* → packages/sentra/*
packages/clinical/* → packages/sentra/*

packages/*          → apps/*
```

---

## Consequences

**Positive:**
- Architectural intent is immediately visible in folder structure
- AI agents can classify and place new packages correctly without Chief intervention
- Boundary enforcement prevents dependency inversion at lint time
- Crown-jewel IP is visually and structurally separated from utility code
- Prepares the taxonomy for future extraction of `packages/sentra/*` to `mother-of-sentra`

**Negative / Trade-offs:**
- Import paths in config files (tsconfig, pnpm-workspace) require one-time update
- Relative `extends` paths in package-level tsconfig files may need updating
- CI/CD path globs must be verified after migration

**Neutral:**
- `pnpm-lock.yaml` will be regenerated by `pnpm install` — this is expected and not a logic change

---

## Enforcement Plan

**Phase 1 — Structural (this migration):**
Folder taxonomy enforced by directory layout.

**Phase 2 — ESLint (Phase 5 of ABYSS-REPO-STRUCTURE-001):**
`packages/tooling/config-eslint` updated with forbidden import patterns to block architectural inversion at lint time.

**Phase 3 — Future (Nx tags, if adopted):**
If Nx is introduced, add tags (`scope:sentra`, `scope:platform`, etc.) and constraints in `nx.json` to enforce dependency direction at graph level.

**Agent Governance:**
`AGENTS.md` updated with explicit rule: agents must not create packages directly under `packages/*`. If classification is unclear, stop and request Chief decision.
