# AADI V2 — Two-Layer Work Breakdown Structure
## WBS + Claude/Codex Allocation · 2026-04-27

---

## Context

AADI V2 adalah program upgrade `@sentra/nada` dari safety-triage engine
(maturity 50/100) menjadi native diagnostic reasoning engine (target 88–92/100).

Current Symphony kuat di safety alerting (NEWS2, vital alerts, PE suspect,
anaphylaxis, DDI, traffic-light), tetapi diagnosis masih bergantung pada
`diagnosisCandidates` dari luar, engine status eksplisit `degraded`, dan
confidence global stuck di `insufficient_data`.

AADI V2 membangun native reasoning spine di dalam SYMPHONY tanpa memindahkan
authority ke sentra-rag, retrieval lane lain, Dashboard, atau Assist. Semua perubahan
bersifat additive-first terhadap existing engine.

### SPRINT 1 — Clinical Facts Spine

**Exit criteria:** ClinicalFacts terbentuk dari existing engines, syndrome classifier
berjalan, parity lama tetap hijau.

---

#### Task 1 — Extend Shared AADI V2 Contracts

- **Owner:** Claude-lead (70%) / Codex assist (30%)
- **Deskripsi:** Tambah tipe baru ke `packages/shared/shared-types/src/symphony.ts`:
  `SymphonyClinicalDisposition`, `SymphonyClinicalFact`, `SymphonyReasoningEvidence`,
  `SymphonyDiagnosticHypothesis`, `SymphonyShadowComparison`. Tidak mengubah tipe lama.
- **Deliverable:** Types extended di shared-types, semua consumer tetap compile.
- **Effort:** ~2 hari
- **Dependency:** None
- **Claude scope:** Type design, clinical contract correctness, interface architecture
- **Codex scope:** Boilerplate type file creation, export wiring
- **Verification:** `pnpm typecheck`, `pnpm lint`

---

#### Task 2 — Build Clinical Facts from Existing Engines

- **Owner:** Claude-heavy (80%) / Codex assist (20%)
- **Deskripsi:** Buat `packages/sentra/sentra-nada/src/engine/clinical-facts.ts` yang mengonsumsi
  output dari 20 mandatory reuse targets (snapshot, symptom signals, clinical patterns,
  NEWS2, trajectory, dll.) dan menghasilkan `SymphonyClinicalFact[]`. Tidak menulis ulang
  logic existing — wrap dan reuse.
- **Deliverable:** `clinical-facts.ts` verified reuse `SymphonyClinicalSnapshot`,
  `detectSymphonySymptomSignals()`, `evaluateClinicalPatterns()`, semua parity hijau.
- **Effort:** ~3 hari
- **Dependency:** Task 1
- **Claude scope:** Reuse mapping decisions, integration architecture, data quality signals
- **Codex scope:** Normalization helper scaffolding, basic extractor functions
- **Verification:** `pnpm test`, `pnpm typecheck`, `runSymphonyParityFixtures()`

---

#### Task 3 — Add Syndrome Classification

- **Owner:** Claude-lead (75%) / Codex assist (25%)
- **Deskripsi:** Buat `packages/sentra/sentra-nada/src/engine/syndrome-classifier.ts` yang
  memetakan `SymphonyClinicalFact[]` ke syndrome family canonical Phase 1.
  Initial families harus mengikuti refined implementation plan:
  `acute_febrile_syndrome`, `acute_respiratory_syndrome`,
  `acute_cardiometabolic_syndrome`.
- **Deliverable:** `syndrome-classifier.ts`, test failing → passing, classifier exported,
  taxonomy syndrome tetap sinkron dengan canonical execution plan.
- **Effort:** ~2 hari
- **Dependency:** Task 2
- **Claude scope:** Syndrome mapping clinical logic untuk febrile / respiratory /
  cardiometabolic rules
- **Codex scope:** File structure, basic pattern matching boilerplate
- **Verification:** `pnpm test`, `pnpm typecheck`

### SPRINT 2 — Native Reasoning Core

**Exit criteria:** Native differential berjalan, arbiter mempertahankan safety dominance,
assess.ts terintegrasi dengan feature flag, V1 default tidak rusak.

---

#### Task 4 — Create Diagnosis Packs and Native Differential

- **Owner:** Claude-heavy (80%) / Codex assist (20%)
- **Deskripsi:** Buat `diagnosis-packs.ts` dan `native-differential.ts` sesuai
  canonical execution plan. Initial packs Phase 1 harus tetap:
  `pack-pneumonia`, `pack-sepsis`, dan `pack-htn-crisis`, lalu menghasilkan
  `SymphonyDiagnosticHypothesis[]` tanpa bergantung pada kandidat eksternal.
  Indonesia-first packs seperti dengue / TB suspect / preeclampsia tetap
  visible sebagai roadmap lanjutan, bukan prioritas initial pack Phase 1.
- **Deliverable:** 3 canonical packs Phase 1 (`pneumonia`, `sepsis`,
  `htn-crisis`), differential engine, must-not-miss support,
  no high-confidence default.
- **Effort:** ~4 hari (task terbesar)
- **Dependency:** Task 3
- **Claude scope:** Pack clinical logic untuk pneumonia / sepsis /
  hypertensive-crisis context, must-not-miss rules, confidence calibration
- **Codex scope:** Pack registry boilerplate, file scaffolding, pack index exports
- **Verification:** `pnpm test`, `pnpm typecheck`, feature coverage matrix updated

---

#### Task 5 — Add Reasoning Arbiter

- **Owner:** Claude-dominant (85%) / Codex assist (15%)
- **Deskripsi:** Buat `packages/sentra/sentra-nada/src/engine/reasoning-arbiter.ts` yang
  merekonsiliasi native differential dengan existing safety alerts. Safety selalu
  dominan — critical alert selalu RED, must-not-miss minimal YELLOW.
- **Deliverable:** `reasoning-arbiter.ts`, arbitration rules enforced, semua safety
  dominance rules covered by tests.
- **Effort:** ~3 hari
- **Dependency:** Task 4
- **Claude scope:** Arbitration rules, safety dominance enforcement, conflict resolution,
  confidence adjustment berdasarkan data quality
- **Codex scope:** File scaffolding, basic wiring
- **Verification:** `pnpm test`, `pnpm typecheck`, safety dominance test cases

---

#### Task 6 — Add Explainability and Clinical Disposition Engine

- **Owner:** Claude-lead (70%) / Codex assist (30%)
- **Deskripsi:** Buat `explainability.ts` (per-diagnosis rationale, missing data prompts,
  evidence refs) dan `confidence-engine.ts` (clinical disposition: ok, requires_review,
  insufficient_data, degraded). Tidak call LLM, tidak fabricate evidence.
- **Deliverable:** Explanation builder, clinician summary builder, confidence engine,
  `SymphonyClinicalDisposition` populated correctly.
- **Effort:** ~2 hari
- **Dependency:** Task 5
- **Claude scope:** Clinical disposition logic, explanation structure, missing data prompts
- **Codex scope:** Summary builder templates, builder function boilerplate
- **Verification:** `pnpm test`, `pnpm typecheck`

---

#### Task 7 — Integrate AADI V2 into assess.ts

- **Owner:** Claude-dominant (85%) / Codex assist (15%)
- **Deskripsi:** Wire AADI V2 pipeline ke `assess.ts` secara additive sesuai
  canonical execution plan. Jangan menambahkan dependency pada feature flag baru
  kecuali sudah disetujui terpisah. Integrasi harus mempertahankan output shape
  existing, compatibility dengan `diagnosisSuggestions`, serta safe degraded
  behavior bila native path belum cukup evidence.
- **Deliverable:** `assess.ts` terintegrasi dengan native path, existing output
  compatibility tetap terjaga, dan tidak ada undocumented rollout mechanism
  yang menyimpang dari canonical plan.
- **Effort:** ~2 hari
- **Dependency:** Tasks 1–6
- **Claude scope:** Integration architecture, safe fallback design,
  backward compatibility preservation
- **Codex scope:** Basic wiring, export updates
- **Verification:** `pnpm test`, `pnpm typecheck`, `pnpm lint`, existing test suite hijau

#### Task 8 — Add Shadow Comparison

- **Owner:** Collaborative — Claude (65%) / Codex (35%)

## Non-Negotiable Rules

1. SYMPHONY adalah satu-satunya canonical clinical reasoning engine.
2. Dashboard dan Assist adalah consumer, bukan reasoning authority.
3. Jangan skip feature klinis existing yang relevan.
4. Pertahankan selama migrasi: `diagnosisSuggestions`, `alerts`, `trafficLight`,
   `trajectory`, `quality.auditHints`, action protocol attachment.
5. Setiap task harus update feature coverage matrix sebelum coding.
6. Parity suite wajib tetap hijau setelah setiap task.
7. Jangan menambahkan rollout / feature-flag requirement baru yang belum ada di
   canonical execution plan tanpa approval terpisah.
