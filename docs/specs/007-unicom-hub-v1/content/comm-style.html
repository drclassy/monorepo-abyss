Bagaimana agent-agent akan berbicara melalui UNICOM Hub?

Pola komunikasi ini menentukan protokol dan teknologi yang tepat

LLM Agent A HUB Service B
A → HUB: "diagnose patient X"
HUB → B: route request
B → HUB: "ICD-10: J18.9"
HUB → A: deliver response
⏱ Synchronous · HTTP/gRPC

A — Request / Response

Agent A kirim request, tunggu jawaban dari Agent B via Hub. Seperti API call biasa, tapi dimediasi UNICOM.

✅ Paling familiar    ✅ Mudah di-debug
⚠️ Blocking saat menunggu    ⚠️ Timeout management
HUB (topic router)
📢 A publishes:
"rme.updated"
📥 B subscribes to:
"rme.*"
📥 C subscribes to:
"rme.*"
📥 D subscribes to:
"rme.*"
⚡ Async · Kafka/Redis/NATS

B — Pub / Sub Events

Agent publish event ke topic, agent lain yang subscribe otomatis terima. Tidak ada yang nunggu — fire and forget.

✅ Non-blocking    ✅ Kafka sudah ada!
⚠️ Harder to trace    ⚠️ No guaranteed reply
LLM A HUB LLM B
A ⟷ HUB ⟷ B: persistent WebSocket
A: "Pasien ini kemungkinan..."
B: "Saya setuju, tambahan..."
A: "Tapi ada kontraindikasi..."
🔄 Bidirectional · WebSocket/SSE

C — Streaming / Live Dialogue

Agents berbicara secara real-time melalui Hub — seperti multi-agent conversation. Cocok untuk LLM yang "berdiskusi".

✅ Real-time    ✅ Natural untuk LLM agents
⚠️ Complex state mgmt    ⚠️ Memory intensive

Atau pilih lebih dari satu jika skenario berbeda membutuhkan pola berbeda
