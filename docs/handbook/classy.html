Classy Workflow 2026

Structured execution for high-trust software delivery.

This workflow defines how Classy moves from intent to validated delivery: understand first, plan deliberately, implement with discipline, verify with evidence, then submit for review and close the branch cleanly.

Mandatory Order

Classy command sequence

Default means default. If a task is not purely administrative, follow the sequence unless there is a specific technical reason to compress it.

/classy-using

Set the operating frame

Start from disciplined workflow, not improvisation. Reset to standard operating behavior before touching the task.

/classy-brainstorming

Clarify the problem

Establish the goal, constraints, risks, and decision space. Avoid premature implementation.

/classy-plan

Write the execution path

Convert direction into a concrete implementation sequence that can be executed without drift.

/classy-implement

Build in measured steps

Implement with test-aware discipline, small deltas, and visible control over behavior change.

/classy-review-in

Process review critically

Treat review feedback as input to verify, not commands to obey blindly. Apply only what is technically sound.

/classy-verify

Produce evidence

Run the relevant checks and inspect results. No completion claim exists without verification evidence.

/classy-review-out

Request review only when ready

Submit changes after they are stable and already validated, so reviewers spend time on quality, not cleanup.

/classy-finish

Close the branch cleanly

Finish with merge readiness, branch hygiene, and a clear definition of done.

OpenAI Codex

Best practice 2026

These points synthesize the most relevant official OpenAI guidance for Codex, GPT-5 coding prompts, and code-review workflows.

1. Start local, scoped, and reversible
  • Use Codex in the project directory you actually want it to touch.
  • Prefer local or agent mode when the task requires file reads, commands, and edits.
  • Create Git checkpoints before and after each task so rollback is cheap.
2. Prompt as an engineering system
  • Define the agent role and responsibilities clearly.
  • Specify workflow expectations: plan, execute, test, report.
  • Give concrete tool-use patterns instead of relying on vague autonomy.
3. Optimize for high-confidence deltas
  • Ask for minimal, precise changes when operating in an established codebase.
  • Use prompts that constrain scope, expected files, and verification method.
  • Prefer one well-bounded task over broad mixed objectives.
4. Require evidence, not narrative
  • Testing and validation should be explicit parts of the task.
  • Review diffs and outputs; do not trust edit success messages alone.
  • Use build, test, and smoke checks appropriate to the repo.
5. Use structured progress on long tasks
  • Plan complex work into sub-tasks before execution.
  • Use progress tracking or TODO structure to reduce skipped steps.
  • Reserve detailed preambles for meaningful decisions, not every trivial action.
6. Review before merge, even with automation
  • Codex cloud can propose PR-ready changes, but human review remains mandatory.
  • Use PR review, required checks, and code-owner policy where available.
  • Treat AI review as acceleration, not a replacement for engineering judgment.
Operating Rules

Enforcement intent

The workflow exists to protect delivery quality, review speed, branch stability, and decision clarity.

Mandatory discipline
  • Understand the problem before proposing code.
  • Plan before substantial implementation.
  • Verify before completion or review requests.
  • Use short-lived branches and merge through pull requests.
Use judgment carefully
  • Small tasks may compress steps, but should not skip core reasoning.
  • Pure investigation may start with debugging, then return to the main flow.
  • Incoming review can be challenged if it conflicts with actual codebase reality.
Do not allow
  • Blind implementation from review comments.
  • Completion claims without verification evidence.
  • Long-lived drift between plan and code.
  • Direct branch closure without review readiness.
Expected outcome A Classy task should end with a clear chain of reasoning, bounded changes, observable verification, and branch-level readiness for review or merge.
